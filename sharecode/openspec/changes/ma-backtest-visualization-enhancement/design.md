## Context

当前项目中已有回测与图表相关脚本，但策略逻辑、回测计算和可视化输出耦合度较高，缺少统一数据契约。此次变更需要在不破坏现有分析流程的前提下，补齐“均线策略回测 + 可视化”完整链路，包括信号生成、仓位序列、收益计算、绩效指标和多图输出。

约束：
- 优先复用仓库已有数据格式与脚本组织方式，减少迁移成本。
- 输出结果需可复现（同输入和参数得到同结果）。
- 图像输出需支持回测解释场景（策略行为 + 结果表现）。

## Goals / Non-Goals

**Goals:**
- 提供参数化的均线策略回测引擎（短均线、长均线、手续费、滑点、初始资金）。
- 统一结果数据结构，至少包含：时间序列持仓、逐期收益、累计净值、交易记录、核心指标。
- 提供可视化模块，输出价格/均线/交易点、持仓线、累计收益（或净值）图。
- 提供单一执行入口（脚本或命令）用于一键回测并落盘结果。

**Non-Goals:**
- 不在本次引入多因子组合、择时择券框架或组合优化器。
- 不引入实时交易执行或实盘下单能力。
- 不实现 Web UI，仅提供文件级图像与数据输出。

## Decisions

1. 分层实现：`strategy`（信号）-> `backtest`（持仓/收益）-> `reporting`（指标/图表）
   - Rationale: 降低耦合，便于复用与后续扩展为其他策略。
   - Alternative: 单脚本串行实现。未采用因测试和维护成本高。

2. 使用标准化结果对象（DataFrame + metadata dict）
   - Rationale: 兼容现有 Python 数据分析生态，便于导出 CSV 与绘图。
   - Alternative: 仅返回图像或松散字典。未采用因下游复用性差。

3. 可视化采用多子图方案（价格与信号 / 持仓 / 收益）
   - Rationale: 在单图中同时呈现策略过程和结果，方便回测复盘。
   - Alternative: 输出多张独立图。可作为后续扩展，但首版优先单图联动。

4. 交易假设采用“信号次日生效”并显式可配置
   - Rationale: 避免未来函数，且适配日线回测常见实践。
   - Alternative: 当日收盘成交。可能高估策略表现，默认不采用。

5. 入口参数与输出路径显式化
   - Rationale: 便于自动化脚本调用与批量实验，减少手动改代码。

## Risks / Trade-offs

- [Risk] 不同数据源字段命名不统一导致回测失败 -> Mitigation: 在入口层进行字段映射与必填字段校验。
- [Risk] 均线窗口较大时，前期样本不足影响信号稳定性 -> Mitigation: 明确 warmup 区间并在结果中标记不可交易区间。
- [Risk] 图像信息过多导致可读性差 -> Mitigation: 控制默认图层，提供可选开关输出明细标注。
- [Risk] 交易成本模型过于简化 -> Mitigation: 先支持固定费率模型，后续扩展分段费率/滑点模型。
